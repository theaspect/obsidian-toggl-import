## Context Note

This is an Obsidian desktop plugin. There is no web framework (no React, no Tailwind, no shadcn). All UI is the Obsidian Settings tab rendered entirely via the Obsidian Plugin API (`Setting`, `addDropdown`, `addText`, `addTextArea`, `addToggle`, `setDisabled`, `setHeading`). The "design system" is Obsidian's own stylesheet — the plugin inherits all typography, spacing, and color tokens from the active Obsidian theme. No custom CSS is introduced.

## Spacing Scale

Obsidian controls all spacing via its own stylesheet. The plugin does not declare spacing tokens.

The only spacing that matters to the executor is the **rendering contract**: settings fields appear in the order they are declared in `display()`. No custom margin or padding is added.

Field render order for template mode:

1. Toggl API token (text, password)
2. Test connection (button)
3. Output format (dropdown — gains third option "Custom template")
4. Template string field (textarea — shown only when `outputFormat === 'template'`)
5. Delimiter field (text — shown only when `outputFormat === 'plaintext'`)
6. Sort order (dropdown)
7. Columns heading
8. Column toggles × 5 (disabled/grayed when `outputFormat === 'template'`)

Source: `src/settings.ts` existing order + D-05, D-06, D-07 from CONTEXT.md.

Exceptions: none — no touch targets, no icon-only buttons.

## Typography

All typography is inherited from Obsidian theme. The plugin declares no custom font sizes or weights.

The executor must not add inline styles for text. Use only Obsidian `Setting` API methods:
- `.setName(string)` — setting label (Obsidian controls weight and size)
- `.setDesc(string)` — setting description / helper text (Obsidian controls muted styling)
- `.setHeading()` — section heading (Obsidian controls heading weight)

No custom typography tokens to declare.

## Component Inventory

These are the Obsidian API components the executor will use for new UI in this phase:

| Component | API Call | Phase 10 Usage |
|-----------|----------|----------------|
| Dropdown option | `drop.addOption('template', 'Custom template')` | Third option in Output format dropdown |

| Toggle disable | `toggle.setDisabled(isTemplate)` | Applied to all 5 column toggles when template mode active |

### Template String Field (new — Claude's discretion resolved)

Use `addTextArea`, not `addText`. Templates can be moderately long (e.g. `${description} | ${project} | ${start} (${duration})`); a textarea is more comfortable to edit than a single-line input.

```
Setting:
  Name:        "Template"
  Description: "Available: description, start, duration, tags, project"
  Control:     addTextArea
  Placeholder: "e.g. ${description} (${duration})"
  Value:       settings.templateString
  onChange:    save settings.templateString
```

Source: D-06 from CONTEXT.md + RESEARCH.md Pitfall 4 recommendation.

### Output Format Dropdown (extend existing)

Extend the existing dropdown. The new option is the third entry:

```
addOption('table',     'Markdown table')   // existing
addOption('plaintext', 'Plain text')       // existing
addOption('template',  'Custom template')  // NEW — D-05
```

Cast in `onChange` must be updated to `value as 'table' | 'plaintext' | 'template'`.

### Column Toggle Disable (extend existing loop)

In the existing `for (const col of columns)` loop, compute `isTemplate` before the loop and pass it to each toggle:

```
const isTemplate = this.plugin.settings.outputFormat === 'template';
toggle.setDisabled(isTemplate)
```

Column toggles remain visible — only disabled, not removed from DOM. Source: D-07.

## Interaction States

| State | Trigger | Visual Result |
|-------|---------|---------------|
| Template field visible | `outputFormat === 'template'` | Textarea field rendered below Output format dropdown via `this.display()` re-render |
| Template field hidden | `outputFormat !== 'template'` | Field not rendered in DOM (same pattern as Delimiter field for plaintext mode) |
| Column toggles disabled | `outputFormat === 'template'` | All 5 toggles appear grayed; `setDisabled(true)` applied; they are visible but inactive |
| Column toggles enabled | `outputFormat !== 'template'` | All 5 toggles interactive; `setDisabled(false)` |
| Broken template | `new Function` throws `SyntaxError` | Fallback renderer returns raw template string; no Notice or error shown |
| Unknown variable | `${foo}` in template | Literal `${foo}` appears in inserted note output; no UI feedback |

All state transitions triggered by `outputFormat` dropdown `onChange` → `this.display()` re-render. This pattern is already wired and verified in `src/settings.ts` lines 64-68.
