# Phase 2 — UI Design Contract

> Visual and interaction contract for the Settings phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Context:** This is an Obsidian desktop plugin. The UI surface is the Obsidian Settings panel — not a web app, not a custom DOM view. All visual output is governed by the Obsidian theme system. There is no custom CSS in this phase; `styles.css` is empty. The "design system" is the Obsidian `Setting` API.

## Spacing Scale

Obsidian manages all spacing internally via the `Setting` component. Custom spacing values are not declared for this phase because no custom CSS is written.

The mapping below documents the conceptual grid that `Setting` renders by default (for executor awareness):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing |
| md | 16px | Default element spacing (Setting row padding) |
| lg | 24px | Section padding |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions: none — all spacing is Obsidian-managed. Do NOT add custom `margin` or `padding` CSS to `styles.css` in this phase.

## Typography

Obsidian theme typography applies throughout. No font overrides. The executor introduces zero weight declarations — all weights are Obsidian theme-controlled and the developer must not override them with custom CSS.

Two weight tiers are recognized for semantic purposes only. The exact rendered weight for each tier is determined by the active Obsidian theme, not by plugin code.

| Role | Size | Weight Tier | Line Height |
|------|------|-------------|-------------|
| Setting name (label) | Obsidian default (~14px) | emphasized (theme-controlled) | 1.4 |
| Setting description | Obsidian default (~12px) | regular (theme-controlled) | 1.5 |
| Section heading (`setHeading()`) | Obsidian default (~13px, styled as subheading) | emphasized (theme-controlled) | 1.2 |
| Input placeholder | Obsidian default (~14px) | regular (theme-controlled) | 1.4 |

**Weight tiers:** `regular` maps to body/description/placeholder roles; `emphasized` maps to names and headings. Obsidian themes typically render regular at 400 and emphasized anywhere from 500–600 — the plugin code never specifies a numeric weight.

**Source:** Obsidian theme system — executor must not override these with custom CSS.

## Layout Order

Settings render top-to-bottom in `display()`. This is the required visual order:

1. **API token** field (masked, `type="password"`) with inline warning as `setDesc()`
2. **Output format** dropdown ("Markdown table" / "Plain text")
3. **Delimiter** field — conditional, appears immediately below Output format when `plaintext` is selected; hidden when `table` is selected
4. **Columns** section heading (`setHeading()`)
5. **Description** toggle
6. **Start time** toggle
7. **Duration** toggle
8. **Tags** toggle
9. **Project** toggle

**Source:** CONTEXT.md D-05, D-06, D-07; RESEARCH.md Pattern 3 (conditional re-render), Pattern 4 (section heading), Pattern 5 (toggles).

## Interaction Contract

| Interaction | Trigger | Response |
|-------------|---------|----------|
| Token typed | `onChange` on text component | Mutate `plugin.settings.apiToken`, call `saveSettings()` — no validation in this phase |
| Token field empty | User clears token | Silent — no error feedback; Phase 5 handles validation |
| Format changed to "Plain text" | Dropdown `onChange` | Mutate `outputFormat`, call `this.display()` then `saveSettings()` — delimiter field appears |
| Format changed to "Markdown table" | Dropdown `onChange` | Mutate `outputFormat`, call `this.display()` then `saveSettings()` — delimiter field disappears |
| Delimiter typed | `onChange` on text component | Mutate `plugin.settings.delimiter`, call `saveSettings()` |
| Column toggled | `onChange` on toggle component | Mutate `plugin.settings.columns[key]`, call `saveSettings()` |
| Settings tab opened | `display()` called | `containerEl.empty()` then full re-render from current `plugin.settings` state |

**Source:** RESEARCH.md §Architecture Patterns (Patterns 2–5), Pitfall 1 (type cast), Pitfall 2 (re-render order).
