Create `.github/workflows/release.yml` with the following exact content (per D-01, D-02, D-03):

```yaml
name: Release Obsidian plugin

on:
  push:
    tags:
      - "[0-9]+.[0-9]+.[0-9]+*"

env:
  PLUGIN_NAME: obsidian-toggl-import

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: write

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "22"

      - name: Build plugin
        run: |
          npm install
          npm run build

      - name: Create release
        uses: softprops/action-gh-release@v2
        with:
          generate_release_notes: true
          files: |
            main.js
            manifest.json
            styles.css
```

Implementation notes:
- Tag pattern `[0-9]+.[0-9]+.[0-9]+*` matches bare semver tags like `1.0.0` and `1.0.1` — NO `v` prefix (per D-01; Obsidian's community validator requires the release tag to match manifest.json `version` exactly)
- `permissions: contents: write` is REQUIRED at job level — without it, softprops/action-gh-release fails with HTTP 403 (per research Pitfall 2)
- `generate_release_notes: true` satisfies D-02 — GitHub generates notes from commits since last tag automatically
- `npm install && npm run build` only — no separate `tsc --noEmit` step (per D-03; `npm run build` already includes type-checking)
- Use `softprops/action-gh-release@v2` — the deprecated two-action pattern (`actions/create-release` + `actions/upload-release-asset`) MUST NOT be used
- Node 22 LTS used in CI (project requires >=18)
- `main.js` is gitignored — the build step produces it; styles.css is already committed as an empty file

**2a. Create `version-bump.mjs`** — verbatim from official obsidian-sample-plugin with the key-check bug fix applied (per D-04 and research Pitfall 4):

```javascript
import { readFileSync, writeFileSync } from "fs";

const targetVersion = process.env.npm_package_version;

// read minAppVersion from manifest.json and bump version to target version
const manifest = JSON.parse(readFileSync("manifest.json", "utf8"));
const { minAppVersion } = manifest;
manifest.version = targetVersion;
writeFileSync("manifest.json", JSON.stringify(manifest, null, "\t"));

// update versions.json with target version and minAppVersion from manifest.json
// Guard checks the version KEY (not minAppVersion values) so patch bumps that share
// the same minAppVersion still get recorded. This deviates from the official sample
// plugin's Object.values check, which skips entries when minAppVersion doesn't change.
const versions = JSON.parse(readFileSync("versions.json", "utf8"));
if (!versions[targetVersion]) {
	versions[targetVersion] = minAppVersion;
	writeFileSync("versions.json", JSON.stringify(versions, null, "\t"));
}
```

**2b. Update `package.json`** — add the `"version"` lifecycle script to the existing `scripts` block (per D-04). The final scripts section must be:

```json
"scripts": {
    "dev": "node esbuild.config.mjs",
    "build": "tsc --noEmit && node esbuild.config.mjs production",
    "test": "vitest run",
    "version": "node version-bump.mjs && git add manifest.json versions.json"

Do NOT modify any other field in package.json.

**2c. Create `.npmrc`** — enforce no-`v`-prefix on tags produced by `npm version`:

```

This ensures `npm version patch` creates a `1.0.1` git tag (not `v1.0.1`), which must match `manifest.json`'s `version` field for Obsidian's community plugin submission validator.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| tag push → workflow trigger | An authenticated GitHub user pushes a tag; the Actions runner executes with `GITHUB_TOKEN` |
| workflow → GitHub release | The runner creates a release and uploads assets using the scoped `GITHUB_TOKEN` |
| npm version → git tag | Developer machine creates a commit and tag that triggers the workflow |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | `softprops/action-gh-release@v2` third-party action | mitigate | Pin to `@v2` (stable tag, actively maintained). Do not use `@main` or `@latest`. Re-pin to a SHA if stricter supply-chain control is required in the future. |
| T-06-02 | Elevation of Privilege | `permissions: contents: write` | mitigate | Grant only `contents: write` at the job level — minimum required for release creation. Do not use `write-all`. No other permissions granted. |
| T-06-03 | Information Disclosure | `GITHUB_TOKEN` in workflow logs | accept | `GITHUB_TOKEN` is never logged explicitly. GitHub Actions masks it automatically. Default token is scoped to repo only; no additional secrets are added. Low risk. |
| T-06-04 | Tampering | Fork PRs triggering workflow | accept | Tag-triggered workflows only run for tags pushed to the main repo by authenticated users. Fork PRs cannot push tags to the upstream repo. No mitigation needed. |
| T-06-05 | Tampering | `version-bump.mjs` modifying version files | mitigate | Script reads `npm_package_version` (set by npm — not user-supplied argv). No user input is parsed. writeFileSync writes only manifest.json and versions.json — no arbitrary file paths. |
| T-06-06 | Denial of Service | Triggering workflow on every tag push | accept | The tag pattern `[0-9]+.[0-9]+.[0-9]+*` restricts triggers to semver-format tags only. Non-semver tags (e.g. branch protection tags) do not trigger the workflow. Acceptable for a single-developer plugin project. |
</threat_model>
